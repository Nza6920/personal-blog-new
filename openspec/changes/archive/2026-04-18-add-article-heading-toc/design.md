## Context

当前文章详情页直接输出正文 HTML/Markdown 渲染结果，没有章节级导航。正文来源同时支持 Markdown 和 HTML，两种内容最终都在 `resources/views/topics/index.blade.php` 的 `.topic-body` 容器中渲染，因此目录能力需要兼容两类正文来源，并尽量不引入额外后端存储字段或编辑流程。

页面已经包含少量 jQuery 脚本用于主题切换和图片预览，因此新增目录交互应优先复用现有页面脚本能力，避免为单页功能引入新的前端依赖或重构整页资源加载方式。

## Goals / Non-Goals

**Goals:**
- 自动识别正文中的一级标题并生成目录，无需作者手工维护目录数据。
- 为每个目录项提供可直接跳转的锚点链接，并支持刷新后继续通过 URL hash 定位章节。
- 在用户滚动正文时高亮当前章节，提升长文阅读定位能力。
- 当正文不存在一级标题时优雅降级，不显示无效目录。

**Non-Goals:**
- 不处理二级及以下标题的多层级目录。
- 不改动文章编辑器或后台录入方式。
- 不引入独立前端框架、构建链或第三方目录插件。

## Decisions

### 1. 在客户端基于已渲染 DOM 生成目录

页面正文已经在服务端完成 Markdown/HTML 渲染，客户端只需要在 `.topic-body` 中扫描 `h1` 节点即可生成目录。这样可以同时兼容 Markdown 和 HTML 两种正文来源，且不需要在控制器或模型层增加正文解析逻辑。

备选方案：
- 服务端解析正文并下发目录数据：可以减少前端扫描，但需要分别处理 Markdown 与 HTML 内容，复杂度更高，且会把表现层需求下沉到后端。
- 作者手工维护目录：实现简单，但会增加后台使用成本，且容易与正文结构不一致。

### 2. 为一级标题注入稳定且唯一的 `id`

目录跳转依赖标题锚点。实现时应在页面加载后为每个 `h1` 生成 slug 风格 `id`，并在重复标题时追加递增后缀以保证唯一性。若标题已有 `id`，则优先复用，避免覆盖已有链接。

备选方案：
- 始终重写标题 `id`：会破坏正文作者显式设置的锚点。
- 使用纯序号作为 `id`：实现容易，但 URL 可读性差，也不利于调试与外部引用。

### 3. 使用 `IntersectionObserver` 同步当前章节状态

滚动高亮采用 `IntersectionObserver` 监听标题进入视口的状态，并以距离顶部最近的可见一级标题作为当前章节。该方案比频繁绑定 `scroll` 事件更轻量，且更容易控制切换阈值。

备选方案：
- 原生 `scroll` + `getBoundingClientRect`：兼容性足够，但实现更冗长，持续滚动时计算频率更高。
- 只在点击目录后设置激活态：无法反映自然滚动场景，不满足需求。

### 4. 目录区域仅在存在一级标题时渲染

目录容器可先在模板中预留结构并默认隐藏，客户端扫描到至少一个 `h1` 后再填充并显示。这样可以保持页面结构稳定，同时避免无标题文章出现空白占位。

## Risks / Trade-offs

- [Markdown 或 HTML 内容中存在重复一级标题] → 通过检测已使用 slug 并追加序号后缀保证锚点唯一。
- [长标题生成的锚点包含不可读字符] → 生成 slug 时清理空白和特殊字符，必要时回退到顺序编号。
- [视口较高或页面顶部大图导致当前章节切换不稳定] → 为 `IntersectionObserver` 设置合适的 `rootMargin`，让高亮逻辑以接近正文可阅读区域的位置为准。
- [目录交互依赖 JavaScript] → 无脚本环境下正文仍可正常阅读，只是不会显示增强目录。

## Migration Plan

该变更仅影响文章详情页前端展示，无需数据迁移。上线后可直接发布；若需要回滚，只需移除目录相关模板、样式和脚本。

## Open Questions

- 目录在移动端采用顶部折叠还是正文前内联展示，可在实现阶段根据现有页面宽度决定具体样式，但不影响本次能力边界。
